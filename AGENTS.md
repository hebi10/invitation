# AGENTS.md

## 문서 목적
- 이 문서는 이 프로젝트에서 작업하는 AI(Codex, Claude, Copilot 포함)가 동일한 기준으로 일관되게 작업하기 위한 공통 실행 규칙이다.
- 모든 작업은 안정적인 수정, 기존 동작 보존, 유지보수성 향상을 우선한다.
- "3. 작업 실행 루프"의 작업 순서를 지키며 작업한다.

## 문서 우선순위
- 공통 규칙은 이 문서를 최우선으로 따른다.
- 도구별 전용 문서(Claude, Codex, Copilot)는 이 문서를 보완하는 용도로만 사용한다.
- 전용 문서와 충돌하면 이 문서를 우선한다.

## 1. 프로젝트 개요
- 서비스 성격: 모바일 청첩장 + 추억(메모리) 페이지 운영 서비스
- 핵심 사용자 흐름
  - 방문자: 청첩장 공개 페이지 조회, 방명록 작성
  - 관리자: 페이지/이미지/댓글/노출기간/비밀번호 관리
  - 고객: 페이지별 비밀번호로 편집기 진입 후 내용 수정
- 주요 라우팅 흐름
  - 공개 청첩장: 슬러그 기반 페이지, 테마(감성형/심플형) 분기
  - 메모리 페이지: 슬러그 기반 공개/비공개 노출
  - 운영 페이지: 관리자 화면, 고객 편집기 화면
- 데이터 흐름 원칙
  - 청첩장 데이터는 Firestore 우선 조회
  - 로컬 또는 권한 제약 시 샘플 데이터 fallback 허용
  - 댓글은 신규 guestbooks 구조 우선, 레거시 구조 fallback 유지

## 2. 기술 스택
- HTML, CSS, JavaScript
- React, Next.js(App Router)
- TypeScript
- Firebase Auth, Firestore, Firebase Storage
- Playwright, ESLint, TypeScript type check
- 라이브러리 도입 원칙
  - 새 라이브러리 추가 전에는 기존 구조로 해결 가능한지 먼저 검토한다.
  - 현재 프로젝트에 미도입 상태인 라이브러리는 필요가 명확할 때만 도입을 제안한다.
  - 이미 도입된 라이브러리는 해당 라이브러리의 권장 패턴을 따른다.
  - 도입 이후 해당 문서와 README.md파일을 업데이트 한다.

## 2-1. 디렉토리 역할
- `src/app/` — App Router 페이지·레이아웃·API 라우트
- `src/app/*/_components/` — 해당 라우트 전용 UI 컴포넌트
- `src/app/*/_hooks/` — 해당 라우트 전용 커스텀 훅
- `src/services/` — 클라이언트에서 호출하는 데이터 서비스 (Firebase 클라이언트 SDK, fetch 등)
- `src/server/` — 서버 전용 로직 (Firebase Admin SDK, 서버 사이드 서비스)
- `src/lib/` — 공유 설정·정책 (클라이언트/서버 공용 가능)
- `src/utils/` — 순수 유틸리티 함수
- `src/data/` — 시드 데이터·정적 데이터
- `public/` — 정적 에셋

## 3. 작업 실행 루프
모든 작업은 아래 순서를 기본 루프로 따른다.

### 3-1. GROUND
- 수정 전에 관련 API, package.json 스크립트, 파일 코드를 확인하고 그에 맞는 서비스 흐름을 파악한다.
- 기억이나 추측으로 구조를 가정하지 않는다.
- 실제 프로젝트에 존재하는 패턴과 서비스의 흐름이 깨지지 않는 것을 우선 관찰한다.
- 라우트, 서비스, 서버 로직, 권한 경계를 먼저 확인한다.

### 3-2. APPLY
- 관찰한 패턴에 맞춰 구현한다.
- 아래 5가지 불변 제약을 지킨다.
  - 읽기 우선
  - 패턴 준수
  - 정책 보존
  - 최소 변경
  - 스코프 준수

### 3-3. VERIFY
- 수정 후 가능한 범위에서 lint, typecheck, build, 관련 테스트를 수행한다.
- 테스트가 통과하면 README.md 파일을 확인하여 수정 사항을 체크한다.
- 사람의 감각적 확인만으로 완료 처리하지 않는다.
- 검증 명령은 package.json에 정의된 실제 스크립트명을 우선 사용한다.

### 3-4. ADAPT
- 검증 실패 시 바로 원인을 분석하고 GROUND 단계로 돌아간다.
- 같은 원인으로 2회 이상 실패하면 접근 방식을 바꾼다.
- 3회 이상 반복되거나 권한/환경 제약으로 해결 불가능하면 현재 상태, 실패 원인, 남은 리스크를 정리해 보고한다.

## 4. 작업 원칙
- 모든 답변과 설명은 한국어로 작성한다.
- 기존 구조를 먼저 이해한 뒤 수정한다.
- 과도한 리팩터링보다 안정적인 수정 우선으로 작업한다.
- 한 번에 너무 많은 파일을 수정하지 않는다.
- 공통화는 재사용 이득이 명확할 때만 진행한다.
- UI, 비즈니스 로직, 서버 로직이 섞여 있으면 분리 방향을 먼저 제안한다.
- 기존 동작을 깨뜨릴 위험이 있으면 수정 전 위험 요소를 먼저 명시한다.
- 사용하지 않는 파일, 빈 파일, 죽은 코드는 정리 후보로 표시한다.
- 에러 수정 시 원인, 수정 범위, 재발 가능성까지 함께 제시한다.
- 유지보수성과 재사용성을 항상 우선한다.
- 요청하지 않은 범위까지 확장 구현하지 않는다.
- 기존 정책과 권한 모델을 임의로 단순화하지 않는다.

## 5. 파일 수정 원칙
- 요청 범위 내부 파일부터 최소 변경한다.
- 파일명/폴더 구조를 불필요하게 대량 변경하지 않는다.
- 화면 로직은 UI 계층, 데이터 입출력은 서비스 계층, 서버 전용 로직은 서버 계층에 둔다.
- 기존 API 응답 형태와 필수 필드를 함부로 변경하지 않는다.
- 인증/권한 관련 로직은 회귀 위험을 먼저 점검한다.
- 중요한 구조 변경 시 이유와 영향 범위를 반드시 설명한다.
- 수정 대상과 직접 관련된 파일부터 읽고, 필요한 경우에만 인접 파일로 범위를 넓힌다.
- 추상화는 반복이 실제로 확인됐을 때만 도입한다.

## 6. 코드 작성 원칙
- 코드베이스 수정 요청에서는 대상 파일 형식(ts/tsx/js/css 등)에 맞춰 직접 수정 가능한 형태로 제공한다.
- 퍼블리싱 산출물 요청일 때만 HTML, CSS, JS를 구분해서 제공한다.
- 특정 파일 형식을 요청받으면 해당 형식만 제공한다.
- 코드 주석은 최소화한다.
- 설명은 코드 아래에 별도로 작성한다.
- 함수명, 변수명, 컴포넌트명은 역할이 드러나게 작성한다.
- 불필요한 추상화보다 읽기 쉬운 구조를 우선한다.
- 입력값 검증, 예외 처리, 사용자 메시지를 빠뜨리지 않는다.
- 새 파일 생성은 기존 구조로 해결할 수 없는 경우에만 진행한다.
- 관찰된 프로젝트 패턴과 다른 구현을 넣을 때는 이유를 설명한다.

## 7. UI 작업 원칙
- 시맨틱 마크업을 우선한다.
- class 이름은 역할이 드러나게 작성한다.
- 모바일 우선으로 보고 데스크톱 깨짐까지 확인한다.
- 버튼/링크 역할, label 연결, alt, 키보드 접근성을 무시하지 않는다.
- 스타일 수정 시 시각적 결과뿐 아니라 유지보수 구조도 고려한다.
- 디자인 변경이 기능 흐름, 접근성, 반응형 레이아웃에 미치는 영향을 함께 점검한다.

## 8. 상태 관리 / 데이터 처리 원칙
- 서버 데이터는 단일 소스 우선 원칙을 유지한다.
- Firestore 우선 + fallback 구조를 깨지 않는다.
- UI 상태와 서버 상태를 혼합하지 않는다.
- 로딩/성공/실패 상태를 명시적으로 관리한다.
- 저장 시 필수값 검증을 수행한다.
- 날짜, 슬러그, URL, 전화번호 등 포맷 검증을 수행한다.
- 마이그레이션 과도기 데이터는 하위 호환을 유지한다.
- 타입, 시드, 서비스 계층 사이에서 같은 의미의 값이 다르게 해석되지 않도록 확인한다.

## 9. API / 서버 연동 원칙
- App Router 기준으로 API 라우트를 다룬다.
- 요청/응답 계약을 명확히 유지한다.
- 상태 코드는 의도에 맞게 사용한다.
- 관리자 인증 흐름과 고객 편집 세션 흐름을 혼동하지 않는다.
- 외부 API 호출 실패 시 안전한 fallback 또는 명확한 실패 응답을 제공한다.
- 환경 변수 규칙
  - `NEXT_PUBLIC_` 접두사가 붙은 변수만 클라이언트 번들에 포함된다.
  - 서버 전용 변수(Firebase Admin 키, API 시크릿 등)는 `src/server/` 또는 API 라우트에서만 사용한다.
  - 환경 변수를 새로 추가할 때는 공개/서버 구분을 반드시 명시한다.
- 민감 정보는 코드, 로그, 커밋에 노출하지 않는다.
- 사용자용 메시지와 내부 디버깅 로그를 분리한다.
- 이미지 업로드 이중 경로
  - 관리자: 클라이언트에서 Firebase Storage에 직접 업로드 (`src/services/imageService.ts`)
  - 고객 편집기: API 라우트를 경유하여 서버에서 Admin SDK로 업로드 (`src/app/api/` 경유)
  - 이 경계를 혼동하면 권한 모델이 깨지므로 수정 시 어느 경로인지 먼저 확인한다.

## 10. 하네스 계층 원칙
이 프로젝트의 가드레일은 아래 3층으로 구분한다.

### 10-1. Layer 1 — 시스템 레벨 강제
- hooks, guard 스크립트, Git 훅, CI, formatter, lint, typecheck 등 자동 실행 장치
- 위험 명령 차단, 민감 파일 보호, 품질 게이트, 자동 포맷팅은 이 계층이 우선한다.
- 프롬프트를 무시하더라도 이 계층은 우회하지 않는 방향으로 설계한다.

### 10-2. Layer 2 — 공통 규칙 문서
- AGENTS.md, CLAUDE.md, codex.md, copilot instructions 등
- 작업 순서, 프로젝트 컨벤션, 금지 규칙, 검증 기준을 정의한다.
- 시스템 레벨에서 강제되지 않는 판단 기준을 통일한다.

### 10-3. Layer 3 — 역할별 워크플로우
- 구현, 리뷰, 리팩터링, 디버깅, 배포 준비 같은 작업별 절차
- 필요 시 도구별 문서에서 역할 특화 규칙을 추가한다.
- Layer 2와 충돌하면 Layer 2를 우선하고, 보안/품질 관련 강제는 Layer 1을 우선한다.

## 11. 위험 명령 / 민감 파일 원칙
- 대량 스테이징, 강제 푸시, 파괴적 삭제 명령은 기본적으로 금지한다.
- 예시
  - `git add -A`
  - `git add .`
  - `git push --force`
  - `rm -rf`
- `.env`, 인증 키, 서비스 계정 키, `.pem`, 배포 비밀값 등 민감 파일은 임의로 생성·수정·노출하지 않는다.
- 민감 파일 수정이 정말 필요하면 목적, 영향 범위, 노출 위험을 먼저 설명한다.
- 자동 생성 파일이나 빌드 산출물은 소스 수정으로 해결 가능한지 먼저 확인한다.

## 12. 테스트 / 검증 원칙
- 수정 후 lint, type, build, runtime 관점에서 점검한다.
- 검증 명령은 package.json에 정의된 실제 스크립트명을 우선 사용한다.
  - lint: `npm run lint`
  - build: `npm run build`
  - typecheck: 별도 스크립트 없음 → `npx tsc --noEmit` 사용
  - e2e smoke: `npm run test:e2e:smoke`
- 에이전트 환경(샌드박스 등)에서 브라우저·Firebase 연동이 불가능하면 lint/type/build까지를 기본 검증 범위로 한다.
- Playwright 검증은 전체 테스트보다 주요 흐름 smoke 확인을 우선한다.
- 권장 점검 순서
  1. lint
  2. typecheck (`npx tsc --noEmit`)
  3. build
  4. 주요 화면 수동 확인
  5. 필요 시 Playwright smoke
- 변경 파일과 직접 관련된 검증을 우선 실행한다.
- 무엇을 확인했는지 체크리스트로 정리한다.
- 실행하지 못한 검증은 이유를 명시한다.
- 확실하지 않은 내용은 불확실하다고 표시한다.
- 성공 여부만 적지 말고 실패 시 원인과 남은 영향까지 적는다.

## 12-1. 실패 처리 기준
- lint 실패: 스타일/규칙 위반 원인 확인 후 수정
- typecheck 실패: 타입 정의, nullable 처리, import/export 불일치 우선 확인
- build 실패: 동적 라우트, 서버/클라이언트 경계, import 경로, 정적 export 영향 우선 확인
- runtime 이슈: 브라우저 재현 경로와 데이터 조건을 함께 기록
- 해결하지 못한 실패는 숨기지 말고 재현 조건을 남긴다.

## 12-2. 커밋 컨벤션
- 커밋 메시지 형식: `Update {변경 요약 한국어}`
- 한 커밋에 관련 변경만 포함한다.
- 브랜치 전략: main 단일 브랜치 기준, 대규모 작업 시 feature 브랜치 분리 가능

## 13. 운영 주의사항
- 정적 export/배포 방식과 동적 라우트 처리 방식이 충돌하지 않는지 먼저 확인한다.
- 로컬 환경에서 관리자 자격증명 또는 Firebase 설정이 없을 때 fallback 동작을 함부로 제거하지 않는다.
- guestbooks 관련 구조는 마이그레이션 과도기일 수 있으므로 신규 구조 우선 + 레거시 호환 여부를 함께 점검한다.
- 공개 페이지, 관리자 페이지, 고객 편집기 흐름은 서로 다른 권한 모델로 다뤄야 한다.
- 슬러그, 공개 여부, 노출 기간과 관련된 로직은 운영 영향이 크므로 수정 범위를 최소화한다.
- package.json, Firebase 설정, 라우트 구조 변경은 운영 영향이 크므로 변경 전후 동작을 분리해서 설명한다.

## 14. 금지 사항
- 근거 없이 전체 구조를 갈아엎지 않는다.
- 요청하지 않은 라이브러리를 무리하게 추가하지 않는다.
- 기존 파일명을 불필요하게 대량 변경하지 않는다.
- 설명 없이 중요한 구조를 바꾸지 않는다.
- deprecated 가능성이 있는 방식을 대안 없이 유지하지 않는다.
- 인증/권한/보안 관련 코드를 임의로 완화하지 않는다.
- 서비스 계정 키, 비밀값, 민감 정보를 노출하지 않는다.
- 검증 없이 완료 처리하지 않는다.
- 읽지 않은 파일을 읽은 것처럼 단정하지 않는다.
- 추측으로 원인을 확정하지 않는다.

## 15. 작업 완료 전 체크리스트
- 요구사항 범위를 벗어난 변경이 없는가
- 기존 사용자 흐름(공개 조회/관리자/고객 편집기)이 유지되는가
- 로딩/에러/빈 상태 처리가 누락되지 않았는가
- 접근성 기본 항목을 확인했는가
- 반응형 레이아웃 깨짐이 없는가
- API 응답 형태와 상태 코드가 일관적인가
- lint/type/build 수행 여부와 결과를 정리했는가
- 실행하지 못한 검증과 남은 리스크를 명시했는가
- 새 의존성 추가 시 필요 근거와 대안을 함께 제시했는가
- 정리 후보 파일을 표시했는가
- 실패 시 어떤 단계에서 ADAPT를 수행했는가
- 시스템 레벨 가드레일로 처리할 항목과 문서 규칙으로 처리할 항목을 구분했는가

## 16. AI 응답 방식
- 한국어로 간결하고 실무적으로 답변한다.
- 요청이 단순하면 짧게, 복잡하면 구조화해서 답변한다.
- 코드가 포함되면 복사 가능한 코드블록으로 제공한다.
- 코드 설명은 코드 아래에 분리해 작성한다.
- 위험 요소가 있으면 먼저 알리고 안전한 수정안을 제시한다.
- 리뷰 요청 시 버그, 리스크, 회귀 가능성을 우선 보고한다.
- 최종 응답에는 가능하면 아래를 포함한다.
  1. 변경 내용 요약
  2. 영향 범위
  3. 검증 결과
  4. 남은 리스크 또는 후속 제안
