# AGENTS.md

## 1. 기본 원칙
- 모든 답변과 설명은 한국어로 작성한다.
- 일반적인 개발 작업 흐름은 Superpowers 스킬을 따른다.
- 작업 판단은 문서보다 실제 코드, 스크립트, API 흐름을 우선한다.
- 추측으로 구조를 가정하지 않는다.
- 기존 동작 보존, 안정적인 수정, 유지보수성 향상을 우선한다.
- 요청 범위를 넘는 리팩터링과 불필요한 파일 수정은 하지 않는다.
- 이 문서는 사용자 요청 없이 수정하지 않는다.

## 2. 작업 전 확인
- 수정 전 관련 파일, API, 서비스 로직, `package.json` 스크립트를 확인한다.
- 라우트, 서버 로직, 저장소, 권한 경계가 관련된 작업은 실제 흐름을 먼저 확인한다.
- 필요한 경우에만 `docs/`의 관련 문서를 참고한다.
- 관련 문서가 없다는 이유만으로 새 문서를 만들지 않는다.

## 3. 코드 수정 규칙
- 기존 구조, 네이밍, 저장 schema, 권한 정책을 우선 보존한다.
- 변경은 요청 범위 안에서 최소 단위로 수행한다.
- 공개 route, API 응답 구조, 인증/인가 흐름을 임의로 바꾸지 않는다.
- 저장 schema 변경이 필요한 경우 영향 범위와 마이그레이션 필요 여부를 먼저 확인한다.
- 한국어 문구를 수정한 경우 깨짐 여부를 간단히 확인한다.

## 4. 검증 규칙
- 기능 변경 후에는 변경 범위에 맞는 lint, typecheck, test, build를 가능한 범위에서 실행한다.
- 검증 명령은 `package.json`에 정의된 실제 스크립트명을 우선 사용한다.
- 검증하지 못한 항목은 완료 보고에 명확히 남긴다.
- 사람의 감각적 확인만으로 완료 처리하지 않는다.

## 5. 보고 규칙
- 작업 완료 보고에는 변경 요약, 검증 결과, 남은 리스크만 간결하게 정리한다.
- 실패가 발생하면 원인, 현재 상태, 남은 작업을 명확히 적는다.
- 작업이 중간에 끊겼거나 검증 실패/수동 QA/남은 리스크가 있을 때만 인수인계 내용을 별도로 남긴다.

## 6. commit 메시지 규칙
- 사용자가 commit 메시지를 요청하면 이전에 push한 git 내용을 확인한 뒤 이번 변경분 기준으로 작성한다.
