## 1. 문서 목적
- 모든 작업은 안정적인 수정, 기존 동작 보존, 유지보수성 향상을 우선한다.
- 추측보다 실제 코드, 스크립트, 서비스 흐름을 기준으로 판단한다.

## 2. 작업 원칙
- 모든 답변과 설명은 한국어로 작성한다.
- 사용자가 commit 메시지를 요청하면 가능한 범위에서 이전 git 코드를 확인하여 `git fetch`, `git diff @{u}` 요약을 한국어로 작성한다.

## 3. DOCS-HUB 규칙
- 모든 작업 시작 전 반드시 `docs/README.md`를 먼저 확인한다.
- `docs/README.md`는 docs 폴더 메인 허브 문서이며, 관련 문서를 찾는 유일한 시작점으로 사용한다.
- 허브 문서에는 아래 형식으로만 문서를 적는다.
  - `어떤 작업의 관련 문서 : 문서명`
- 작업 중에는 허브에서 찾은 관련 문서만 읽는다.
- 허브에 없는 문서는 먼저 읽지 않는다. 정말 필요하면 새 문서를 만들고 허브에 항목을 추가한 뒤 사용한다.
- 작업 종료 시 이번에 읽은 관련 문서를 작업 결과에 맞게 갱신한다.
- 관련 문서가 없어서 새로 만들었다면 작업 종료 전에 내용과 허브 항목까지 함께 정리한다.

## 4. 작업 실행 루프
### 4-1. GROUND
- 먼저 `docs/README.md`에서 이번 작업과 직접 관련된 문서를 찾는다.
- 필요 시 `docs/ai-harness-handoff.md`를 확인해 최근 작업 맥락과 남은 작업을 파악한다.
- 허브에서 찾은 관련 문서만 읽고, 관련 없는 문서는 읽지 않는다.
- 수정 전 관련 API, package.json 스크립트, 핵심 파일을 확인해 실제 흐름을 파악한다.
- 기억이나 추측으로 구조를 가정하지 않는다.
- 라우트, 서비스, 서버 로직, 권한 경계를 먼저 확인한다.

### 4-2. APPLY
- 기존 구조, 네이밍, 권한 정책을 우선 보존한다.
- 요청 범위를 넘는 리팩터링과 불필요한 파일 수정은 하지 않는다.
- 관찰한 패턴에 맞춰 최소 변경으로 구현한다.
- 신규 문서가 필요하면 docs 허브 규칙에 맞춰 생성하고 관련 작업에서만 사용한다.

### 4-3. VERIFY
- 디자인 수정 제외 기능 변경은 변경 범위에 맞는 lint, typecheck, test, build를 가능한 범위에서 실행한다.
- 검증 명령은 package.json에 정의된 실제 스크립트명을 우선 사용한다.
- 수정이 사용 방법, 구조, 명령, 정책에 영향을 주면 관련 문서를 갱신한다.
- 사람의 감각적 확인만으로 완료 처리하지 않는다.

### 4-4. ADAPT
- 검증 실패 시 원인을 분석하고 GROUND 단계로 돌아간다.
- 같은 원인으로 2회 이상 실패하면 접근 방식을 바꾼다.
- 3회 이상 반복되거나 권한/환경 제약이 있으면 반드시 사용자에게 현재 상태, 실패 원인, 남은 리스크를 정리해 보고한다.
- 한국어가 들어간 부분은 깨지는 경우가 많으니 작업한 코드의 한국어를 간단하게 확인해본다.
- 코드 작업 종료 시 `docs/ai-harness-handoff.md`를 갱신한다.
- `docs/ai-harness-handoff.md`는 항상 60줄 이내로 유지한다. 초과하면 오래된 세부 설명, 중복 문장, 참고 메모부터 줄이고 기능 단위 요약으로 다시 묶는다.

## 5. 인수인계
- `docs/ai-harness-handoff.md`에 commit 메시지, 인수인계, 남은 작업을 정리한다.
- 작업 요약과 남은 작업은 각각 최대 3개 핵심 묶음으로 정리한다. 파일별 나열보다 기능/흐름 단위 요약을 우선한다.
- 작업 작성 내용이 길어지면 최근 작업 기준으로 합치고, 세부 파일 경로·명령어·배경 설명은 다음 작업 판단에 필요한 것만 남긴다.
- 문서 갱신 후 줄 수를 다시 확인하고, 60줄을 넘기면 요약을 더 줄인 뒤 종료한다.
- 만약 문서 내용이 없으면 `docs/ai-harness-handoff.template.md`에서 예시 템플릿을 확인한다.
