# 모바일 청첩장 운영 서비스 포트폴리오 정리

## 1. 프로젝트 한 줄 소개

이 프로젝트는 **모바일 청첩장 제작, 공개, 수정, 운영 관리까지 한 번에 다루는 서비스**입니다.  
단순히 예쁜 청첩장 화면만 만드는 것이 아니라, 실제 운영자가 페이지를 만들고, 고객이 직접 수정하고, 하객이 청첩장을 보고, 이후에는 추억 페이지까지 이어지는 흐름을 하나의 서비스 안에 담았습니다.

---

## 2. 이 문서를 보는 방법

이 문서는 면접관이 아래 내용을 빠르게 이해할 수 있도록 정리한 설명서입니다.

- 이 서비스가 어떤 문제를 해결하는지
- 어떤 사용자가 어떤 화면을 사용하는지
- 화면별 역할이 어떻게 나뉘는지
- 데이터가 어디에 저장되고 어떻게 연결되는지
- 운영 서비스로서 어떤 부분을 직접 설계했는지

캡처 이미지는 각 화면 섹션 아래에 넣으면 됩니다.  
스크린샷을 넣기 쉬우도록 **추천 파일명**과 **추천 캡처 포인트**도 함께 적어 두었습니다.

---

## 3. 왜 만든 서비스인가

결혼식 청첩장은 보통 다음 문제가 반복됩니다.

- 제작 과정이 번거롭다
- 고객이 직접 내용을 수정하기 어렵다
- 운영자가 공개 여부, 비밀번호, 노출 기간을 따로 관리해야 한다
- 청첩장과 이후의 추억 페이지가 분리되어 있다

이 프로젝트는 이 문제를 줄이기 위해 만들었습니다.

- 운영자는 관리자 화면에서 여러 청첩장을 한 번에 관리할 수 있습니다.
- 고객은 자신의 청첩장만 비밀번호로 안전하게 수정할 수 있습니다.
- 하객은 모바일에 맞춘 공개 청첩장을 쉽게 볼 수 있습니다.
- 결혼식 이후에는 같은 slug로 추억 페이지까지 연결할 수 있습니다.

즉, **디자인 템플릿 프로젝트가 아니라 실제 운영형 서비스에 가까운 구조**를 목표로 만들었습니다.

---

## 4. 핵심 사용자

### 4-1. 하객

- 공개 청첩장 페이지 접속
- 예식 정보, 지도, 갤러리, 축의금, 방명록 확인
- 필요 시 카카오 공유

### 4-2. 고객

- 페이지별 비밀번호로 편집기 진입
- 이름, 일정, 장소, 이미지, 문구 수정
- 공개 상태 확인

### 4-3. 관리자

- 새 청첩장 draft 생성
- 공개 여부와 테마 상태 관리
- 고객 비밀번호 관리
- 노출 기간 관리
- 이미지와 방명록 관리
- 추억 페이지까지 연결 운영

---

## 5. 서비스 전체 흐름

## 5-1. 가장 큰 흐름

1. 관리자가 새 청첩장을 생성합니다.
2. 위자드 또는 편집기에서 내용을 채웁니다.
3. Firestore에 청첩장 설정과 공개 상태를 저장합니다.
4. 하객은 `/{slug}/{theme}` 경로로 청첩장을 봅니다.
5. 고객은 `/page-editor/{slug}`에서 직접 수정합니다.
6. 관리자는 `/admin`에서 공개 상태, 비밀번호, 노출 기간, 이미지, 댓글을 운영합니다.
7. 결혼식 후에는 `/memory/{slug}` 추억 페이지로 자연스럽게 확장할 수 있습니다.

## 5-2. 한눈에 보는 구조

```text
관리자
  -> /admin
  -> 새 draft 생성 / 공개 상태 / 노출 기간 / 비밀번호 / 이미지 관리

고객
  -> /page-editor/{slug}
  -> 비밀번호 인증 후 청첩장 수정

하객
  -> /{slug}
  -> /{slug}/{defaultTheme} 로 이동
  -> 공개 청첩장 열람

추억 페이지 방문자
  -> /memory/{slug}
  -> 결혼식 이후 사진, 메시지, 댓글 확인
```

## 6. 주요 화면 정리

이 섹션은 실제 포트폴리오 문서에서 가장 중요합니다.  
면접관은 코드보다 먼저 화면 흐름을 보고 서비스 이해도를 판단하는 경우가 많기 때문입니다.

---

- 서비스 첫 진입 화면
- 관리자 페이지, 판매 페이지, 작업 문서로 이동
- 이 프로젝트가 무엇을 만드는 서비스인지 짧게 소개

### 면접관이 보기 좋은 포인트

- 프로젝트 목적이 분명한지
- 개인 토이 프로젝트가 아니라 운영을 염두에 둔 서비스인지
- 다음 진입 지점이 어디인지 명확한지

### 스크린샷 자리

- 추천 파일명: `screenshots/01-home.png`
- 추천 캡처 포인트: 상단 제목, 설명, 이동 버튼이 한 번에 보이게 캡처

### 한 줄 설명

이 화면은 서비스 소개와 운영 진입점을 함께 보여주는 시작 화면입니다.

---

## 6-2. 관리자 대시보드 `/admin`

### 화면 역할

- 전체 운영의 중심 화면
- 청첩장, 추억 페이지, 이미지, 방명록, 비밀번호, 노출 기간을 탭으로 관리

### 핵심 가치

- 운영자가 여러 페이지를 한 곳에서 관리할 수 있음
- 단순 목록이 아니라 실제 운영에 필요한 상태를 빠르게 판단할 수 있음

### 면접관이 보기 좋은 포인트

- 운영 화면이 실제로 존재하는 서비스 구조라는 점
- 공개 상태, 기간, 댓글, 비밀번호 같은 운영 요소를 분리해서 관리했다는 점

### 스크린샷 자리

- 추천 파일명: `screenshots/02-admin-dashboard.png`
- 추천 캡처 포인트: 상단 요약 카드와 탭 영역이 함께 보이게 캡처

### 한 줄 설명

이 화면은 서비스 운영에 필요한 모든 관리 기능을 한 곳에 모은 관리자 콘솔입니다.

---

## 6-3. 관리자 청첩장 탭 `/admin?tab=pages`

### 화면 역할

- 청첩장 목록 확인
- 공개 여부 확인
- 상품 티어 확인
- `emotional`, `simple` 테마 연결 상태 확인
- 편집기 또는 실제 페이지로 이동

### 왜 중요한가

이 서비스는 페이지를 하나 만드는 것보다, **여러 청첩장을 안정적으로 운영하는 것**이 중요합니다.  
그래서 단순 목록보다 “지금 이 청첩장이 공개 가능한 상태인지”를 빠르게 판단할 수 있게 설계했습니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/03-admin-pages-tab.png`
- 추천 캡처 포인트: 청첩장 카드 목록, 공개 상태, 테마 링크가 잘 보이게 캡처

### 한 줄 설명

운영자는 이 화면에서 페이지별 상태를 보고 즉시 수정이나 점검으로 이어질 수 있습니다.

---

## 6-4. 관리자 비밀번호 탭 `/admin?tab=passwords`

### 화면 역할

- 고객 편집용 비밀번호 저장
- 비밀번호 상태 확인
- 해당 청첩장 편집기로 이동

### 왜 중요한가

고객이 직접 수정할 수 있게 하되, 관리자 권한과 섞이지 않도록 분리한 화면입니다.  
이 구조 덕분에 서비스 운영과 고객 수정 권한을 깔끔하게 나눌 수 있습니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/04-admin-passwords-tab.png`
- 추천 캡처 포인트: 비밀번호 입력 영역과 편집기 이동 버튼이 보이게 캡처

### 한 줄 설명

관리자는 이 화면에서 고객 전용 편집 권한을 페이지별로 안전하게 관리합니다.

---

## 6-5. 관리자 노출 기간 탭 `/admin?tab=periods`

### 화면 역할

- 공개 기간 설정
- 예약 공개, 노출 중, 종료 상태 확인
- 기간 제한 해제 또는 수정

### 왜 중요한가

청첩장은 보통 결혼식 전후에만 공개해야 하기 때문에, 단순 `published`만으로는 부족합니다.  
이 프로젝트는 **공개 여부와 노출 기간을 따로 관리**해서 실제 운영 상황을 반영했습니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/05-admin-periods-tab.png`
- 추천 캡처 포인트: 상태 배지와 기간 설정 UI가 함께 보이게 캡처

### 한 줄 설명

운영자는 이 화면에서 공개 시점을 세밀하게 관리할 수 있습니다.

---

## 6-6. 신규 청첩장 생성 화면 `/page-wizard`

### 화면 역할

- 새 청첩장 draft 시작
- 시작 템플릿 선택
- 기본 URL 후보 입력
- 신랑, 신부 이름 입력
- 생성 후 상세 위자드로 이동

### 왜 중요한가

처음부터 빈 데이터를 손으로 채우는 것이 아니라, **운영자가 빠르게 새 청첩장을 만들 수 있는 시작점**을 제공합니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/06-page-wizard-create.png`
- 추천 캡처 포인트: 템플릿 선택 카드와 생성 입력폼이 함께 보이게 캡처

### 한 줄 설명

운영자는 이 화면에서 새 청첩장 초안을 빠르게 만들고 상세 편집으로 넘어갑니다.

---

## 6-7. 모바일 위자드 편집 화면 `/page-wizard/{slug}`

### 화면 역할

- 단계형 입력
- 모바일 중심 수정 흐름
- 실시간 요약과 단계 검증
- 대표 이미지, 갤러리, 인사말, 계좌, 음악 등 입력

### 왜 중요한가

이 화면은 단순 입력폼이 아니라 **모바일에서 실제로 수정하기 쉬운 편집 경험**을 목표로 만든 화면입니다.

특히 아래 요소가 중요합니다.

- 단계별 이동
- 자동 파생값 정리
- 주소 검색과 지도 연결
- 이미지 업로드와 순서 조정
- 최종 저장 전 요약 검토

### 추천 세부 캡처

- 기본 정보 스텝
- 장소 입력과 지도 확인 스텝
- 갤러리 업로드 스텝
- 추가 정보 스텝
- 최종 확인 스텝

### 스크린샷 자리

- 추천 파일명: `screenshots/07-page-wizard-detail.png`
- 추천 캡처 포인트: 좌측 또는 상단 진행 상태와 현재 입력 스텝이 동시에 보이게 캡처

### 한 줄 설명

모바일 위자드는 운영자가 빠르게 내용을 입력하고, 저장 전에 전체 흐름을 점검할 수 있는 관리자용 입력 화면입니다.

### 왜 중요한가

저장 직후 “정말 제대로 반영되었는지”를 한 번 더 확인할 수 있는 중간 검토 화면입니다.  
운영 관점에서는 이 화면이 실수를 줄여 줍니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/08-page-wizard-result.png`
- 추천 캡처 포인트: 저장 완료 카드와 단계별 요약 영역이 같이 보이게 캡처

### 한 줄 설명

이 화면은 저장 직후 운영자가 최종 검수를 할 수 있게 돕는 확인 화면입니다.

## 6-9. 고객 편집기 시작 화면 `/page-editor`

### 화면 역할

- 기본 편집기용 draft 생성 진입점

### 왜 중요한가

위자드가 모바일 중심이라면, 이 흐름은 보다 일반적인 편집 흐름에 가깝습니다.  
서비스 안에서 생성 방식과 수정 방식이 분리되어 있다는 점을 보여주기 좋습니다.

### 스크린샷 자리

- 추천 파일명: `screenshots/09-page-editor-create.png`
- 추천 캡처 포인트: 시작 설명과 생성 입력 영역이 보이게 캡처

### 한 줄 설명

이 화면은 고객 편집기로 들어가기 전 새 페이지를 만드는 시작점입니다.

---

## 6-10. 고객 편집기 `/page-editor/{slug}`

### 화면 역할

- 고객이 직접 청첩장 내용을 수정하는 화면
- 관리자 또는 페이지별 비밀번호로 진입
- 자동 저장
- 미리보기 테마 전환
- 공개 상태 저장

### 왜 중요한가

이 화면이 있어야 서비스가 단순 제작물이 아니라 **실제 수정 가능한 서비스**가 됩니다.  
고객은 개발자나 운영자를 거치지 않고 직접 내용을 바꿀 수 있습니다.

### 이 화면에서 강조할 점

- 권한이 관리자와 분리되어 있음
- 서버 API를 거쳐 안전하게 저장함
- 입력과 미리보기가 한 흐름으로 이어짐
- 공개 전 최종 점검이 가능함

### 스크린샷 자리

- 추천 파일명: `screenshots/10-page-editor-detail.png`
- 추천 캡처 포인트: 좌측 단계 메뉴와 우측 입력 또는 미리보기 화면이 함께 보이게 캡처

### 한 줄 설명

고객 편집기는 서비스 운영 부담을 줄여 주는 핵심 화면입니다.

---

## 6-11. 공개 청첩장 감성형 `/{slug}/emotional`

### 화면 역할

- 실제 하객이 보는 공개 청첩장
- 사진 중심 레이아웃
- 일정, 장소, 지도, 갤러리, 방명록, 카카오 공유 제공

### 왜 중요한가

이 화면은 사용자에게 보이는 최종 결과물입니다.

- 추천 파일명: `screenshots/11-public-emotional.png`

### 화면 역할

### 한 줄 설명

## 7. 이 서비스의 핵심 기능

### 7-1. 테마 기반 공개 페이지

- `emotional`, `simple` 두 가지 테마 제공
- 같은 데이터로 다른 공개 화면 제공
- 공유 URL은 `/{slug}`에서 기본 테마 경로로 연결

### 7-2. 관리자 운영 기능

- 청첩장 생성과 수정
- 공개 상태 관리
- 노출 기간 관리
- 고객 비밀번호 관리
- 이미지와 방명록 운영

### 7-3. 고객 직접 수정 기능

- 페이지별 비밀번호 인증
- 자동 저장
- 관리자 권한과 분리된 편집 흐름

### 7-4. 운영 친화적 데이터 구조

- Firestore 우선 조회
- 로컬 sample fallback
- 공개 페이지와 운영 페이지의 권한 분리
- 이미지, 댓글, 공개 기간까지 각각 관리

### 7-5. 추억 페이지 확장

- 청첩장과 같은 slug 기반 연결
- 결혼식 이후 사진과 메시지를 별도 페이지로 운영

---

## 8. 기술 구조를 쉬운 말로 설명하면

### 8-1. 프론트엔드

- Next.js App Router 기반
- 공개 페이지, 관리자, 위자드, 편집기, 메모리 페이지를 라우트로 분리
- React 상태 관리와 단계형 입력 UI 구현

### 8-2. 데이터 저장

- Firestore: 청첩장 설정, 공개 상태, 비밀번호, 기간, 댓글 저장
- Firebase Storage: 청첩장 이미지와 메모리 이미지 저장

### 8-3. 서버 역할

- 공개 가능한 청첩장인지 확인
- 고객 편집 세션 검증
- API 경유 저장 처리
- 공개/비공개와 노출 기간 상태 판정

### 8-4. 왜 이 구조가 중요한가

청첩장 서비스는 “예쁜 화면”보다도 아래가 더 중요합니다.

- 누가 수정할 수 있는지
- 언제 공개할지
- 공개 페이지에서 실제로 보여도 되는지
- 운영자가 실수 없이 관리할 수 있는지

이 프로젝트는 그 부분을 코드 구조와 데이터 구조 안에 담으려는 방향으로 설계했습니다.

## 9. 데이터 흐름 요약

## 9-1. 공개 청첩장

1. 사용자가 `/{slug}` 또는 `/{slug}/{theme}`에 접속합니다.
2. 서버가 Firestore에서 해당 청첩장 정보를 먼저 확인합니다.
3. 공개 상태와 노출 기간을 함께 검사합니다.
4. 공개 가능하면 실제 청첩장 데이터를 렌더링합니다.
5. Firestore를 못 쓰는 환경에서는 sample 데이터를 fallback으로 사용합니다.

## 9-2. 고객 편집 저장

1. 고객이 비밀번호로 편집기 인증을 합니다.
2. 서버가 세션 쿠키를 발급합니다.
3. 저장 요청은 `/api/client-editor/*` 경로로 보냅니다.
4. 서버가 세션을 다시 확인한 뒤 Firestore에 저장합니다.

## 9-3. 이미지 업로드

- 관리자: 클라이언트에서 Storage 직접 업로드
- 고객 편집기: API 라우트를 거쳐 서버에서 업로드

이 분리는 권한 모델을 지키기 위해 중요합니다.

---

## 10. 이 프로젝트에서 보여줄 수 있는 기술적 강점

### 10-1. 단순 랜딩 페이지가 아닌 운영형 구조

공개 화면만 있는 프로젝트가 아니라, 관리자와 고객 편집기까지 포함된 서비스 구조입니다.

### 10-2. 권한 분리

- 관리자 권한
- 고객 편집 권한
- 공개 방문자 권한

세 사용자 역할을 분리해서 설계했습니다.

### 10-3. 실제 서비스에 가까운 운영 기능

- 공개 상태
- 노출 기간
- 비밀번호
- 댓글 관리
- 이미지 관리

단순 CRUD를 넘어서 운영 문제를 다뤘습니다.

### 10-4. fallback 고려

Firestore가 없거나 권한이 없는 환경에서도 sample 데이터를 통해 기본 동작을 유지하도록 구성했습니다.

### 10-5. 확장 가능성

청첩장 이후에 추억 페이지까지 자연스럽게 연결되므로, 단발성 페이지가 아니라 서비스 확장성을 보여 줄 수 있습니다.
